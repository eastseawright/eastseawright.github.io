WordPress Cloud Deployment with AWS Lightsail & CDN

Complete WordPress deployment leveraging multiple AWS services including Lightsail, Route 53, Certificate Manager, S3, CloudFront CDN, and IAM. Implemented secure HTTPS, global content delivery, and scalable media storage with automated offloading from the WordPress server.

Project Overview

This project demonstrates a production-ready WordPress deployment using AWS's suite of cloud services. The architecture prioritizes security (HTTPS everywhere), performance (global CDN), scalability (serverless compute and object storage), and cost efficiency (pay-as-you-go model).

Rather than using traditional server hosting, this implementation leverages AWS Lightsail for simplified WordPress deployment, Route 53 for DNS management, CloudFront for global content delivery, S3 for media storage, and Certificate Manager for SSL/TLS certificates. The result is a fast, secure, and highly available website architecture.

Architecture Overview

Compute Layer

AWS Lightsail instance running WordPress on Linux with Bitnami stack for simplified application deployment.

Content Delivery

CloudFront CDN distributing static assets globally from S3 origin for reduced latency and improved user experience.

Storage & Security

S3 bucket for media storage with IAM roles for secure access, SSL/TLS certificates for HTTPS encryption.

DNS Management

Route 53 handling domain registration, hosted zones, and DNS records for seamless traffic routing.

Implementation Phases

01

Infrastructure Setup: Lightsail & Static IP

Deployed WordPress using AWS Lightsail, which provides a simplified interface for serverless application deployment. Chose a Linux operating system with Bitnami's WordPress stack for optimized performance. Assigned a static IP address to the Lightsail instance to ensure consistent DNS resolution—critical for websites since dynamic IPs can change unexpectedly.

Technical Decision: Lightsail over EC2 provided the right balance of simplicity and control for this use case, with predictable pricing and integrated WordPress management through Bitnami.

02

DNS Configuration: Route 53 Domain & Hosted Zones

Registered domain through Route 53 and created a hosted zone for DNS management. Configured multiple DNS records: A records pointing to the Lightsail static IP (both root domain and www subdomain), and a CNAME record for the CDN subdomain pointing to cloudfront.net for content delivery.

Key Records Created:

  • A record: @ → Lightsail static IP
  • A record: www → Lightsail static IP
  • CNAME record: cdn → CloudFront distribution
03

WordPress Configuration & Initial Setup

Accessed WordPress admin panel at domain.com/wp-admin using Bitnami-generated credentials. Changed default login credentials following security best practices. Configured WordPress settings for the custom domain and prepared the site for SSL certificate installation and S3 integration.

04

Security Implementation: SSL/TLS Certificates

Requested SSL certificate through AWS Certificate Manager (ACM) for the domain. Created certificates for three domain variants: the root domain, www subdomain, and cdn subdomain. ACM provides free SSL/TLS certificates with automatic renewal, crucial for maintaining HTTPS without manual intervention.

Why Multiple Certificates: Each subdomain (www, cdn) needs its own certificate validation to enable HTTPS across all entry points to the site and CDN.

05

Storage Architecture: S3 Bucket & IAM Configuration

Created an S3 bucket for WordPress media storage to offload static files from the Lightsail instance. Configured an IAM user with programmatic access to AmazonS3FullAccess policy. Installed WP Offload Media plugin to automatically sync WordPress uploads to S3, with option to remove files from the server after upload to prevent storage bloat.

Edited wp-config.php in the Bitnami WordPress directory to add AWS credentials (Access Key ID and Secret Access Key) for the IAM user, enabling WordPress to authenticate with S3. Applied bucket policy to allow CloudFront to access objects for content delivery.

Architecture Benefit: Separating media storage from the application server improves performance, reduces server load, and provides scalable storage that can grow without affecting the Lightsail instance.

06

Content Delivery: CloudFront CDN Integration

Created a CloudFront distribution with the S3 bucket as the origin. Configured an Origin Access Identity (OAI) to allow CloudFront to securely access S3 without making the bucket publicly accessible. Set the alternate domain name (CNAME) to the cdn subdomain configured in Route 53.

Updated WP Offload Media plugin to use CloudFront as the delivery provider instead of direct S3 URLs. This ensures all media files are served through CloudFront's global edge network, dramatically reducing load times for international visitors and reducing bandwidth costs on the origin server.

Performance Impact: CloudFront caches content at edge locations worldwide, meaning users download images and media from servers geographically close to them rather than making requests all the way back to the origin S3 bucket.

Technical Highlights & Key Decisions

WordPress Plugin Integration: WP Offload Media

Selected and configured WP Offload Media plugin to automatically sync WordPress media uploads to S3 and serve them through CloudFront. Configured the plugin to remove files from the local server after upload, preventing storage bloat on the Lightsail instance while maintaining seamless media management within WordPress admin.

IAM Security Best Practices

Created dedicated IAM user with programmatic access specifically for S3 operations rather than using root credentials. Granted only AmazonS3FullAccess policy following principle of least privilege. Securely configured credentials in wp-config.php rather than storing in plugin settings.

CloudFront Origin Access Identity

Implemented OAI to allow CloudFront to access S3 bucket contents without making the bucket publicly accessible, maintaining security while enabling global content delivery. This prevents direct access to S3 URLs while still serving content through the CDN.

Multi-Service Integration

Successfully integrated six different AWS services (Lightsail, Route 53, ACM, S3, IAM, CloudFront) into a cohesive architecture. Each service plays a specific role while working together seamlessly— demonstrating understanding of how cloud services interconnect in production environments.
